# Часть 2: IoC и DI

## Задание 2.1

1. Как Spring знает, какой бин подставить в `GreetingController`?

Ответ: Spring анализирует тип параметра конструктора GreetingController (GreetingService) и ищет в контейнере бин, реализующий этот интерфейс. Так как есть только одна реализация (EnglishGreetingService), она и подставляется автоматически благодаря autowiring по типу.

2. Что произойдёт, если убрать `@Service`?

Ответ: Spring не создаст бин для EnglishGreetingService, и при запуске приложения возникнет ошибка NoSuchBeanDefinitionException

3. Что произойдёт при двух реализациях `GreetingService` без дополнительной настройки?

Ответ: Spring не сможет определить, какую из двух реализаций внедрить, и выдаст ошибку NoUniqueBeanDefinitionException. Для решения нужно использовать @Primary (указывает основной бин) или @Qualifier (указывает конкретный бин по имени).

## Задание 2.2

1. Какой способ DI предпочтительнее и почему?

Ответ: Предпочтительнее constructor injection, потому что он делает поля final (неизменяемыми), упрощает тестирование (не нужен Spring-контекст)

2. Почему поле при constructor injection может быть `final`?

Ответ: Поле может быть final, потому что оно инициализируется единственный раз — в конструкторе, до завершения создания объекта. final гарантирует, что зависимость не будет изменена после инициализации, что делает бин потокобезопасным.

3. Какой способ труднее всего тестировать без Spring-контекста?

Ответ: Field injection (внедрение через поле) труднее всего тестировать без Spring, потому что зависимости устанавливаются через рефлексию, и для юнит-тестов приходится использовать рефлексию или специальные библиотеки
